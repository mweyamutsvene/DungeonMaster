# SME Feedback — EntityManagement — Round 1
## Verdict: NEEDS_WORK

## Issues (if NEEDS_WORK)
1. The EntityManagement plan does not include the Prisma NPC model/schema work. The plan changes `SessionNPCRecord`, the repository contract, and the Prisma repository adapter, but `SessionNPC` currently only persists a required `statBlock` JSON blob. Without updating `packages/game-server/prisma/schema.prisma`, the repo layer cannot store class-backed NPC fields such as `sheet`, `className`, and `level`, and it cannot preserve a clean exact-one-representation invariant at the database boundary.
2. The NPC repository contract update is incomplete. The plan says it will replace the stat-block-only create contract in `packages/game-server/src/application/repositories/npc-repository.ts`, but it does not address the existing `updateStatBlock()` API, which is stat-block-specific and becomes misleading or invalid once class-backed NPCs exist. That omission also leaves multiple `INPCRepository` implementations out of sync, including the local test doubles in `packages/game-server/src/infrastructure/api/app.test.ts` and `packages/game-server/src/application/services/combat/combat-service-domain.integration.test.ts`, not just `packages/game-server/src/infrastructure/testing/memory-repos.ts`.

## Missing Context
- The planned route test file `packages/game-server/src/infrastructure/api/routes/sessions/session-creatures.test.ts` does not exist today. Current repository-backed route coverage appears to live in broader app/integration tests, so the implementation plan should either create that new focused test file explicitly or list the existing test files that must be updated.
- If the goal includes deterministic scenario support beyond the player CLI example, the mock combat scenario runner in `packages/game-server/scripts/test-harness/scenario-runner.ts` is still stat-block-only for NPC setup and should be tracked somewhere in the broader plan.

## Suggested Changes
1. Add `packages/game-server/prisma/schema.prisma` to the EntityManagement slice and define the persisted NPC representation explicitly. Recommended direction: make the persistence model representation-aware with nullable `statBlock`, nullable `sheet`, nullable `className`, and nullable `level`, then enforce the XOR invariant in application/API code so existing stat-block NPC rows still load cleanly.
2. Expand the repository-contract step so it covers the full `INPCRepository` surface, not only `createInSession`. Either replace `updateStatBlock()` with a representation-aware update API, or explicitly limit/remove stat-block mutation for class-backed NPCs. In the same step, enumerate every known implementation that must change together: `packages/game-server/src/infrastructure/db/npc-repository.ts`, `packages/game-server/src/infrastructure/testing/memory-repos.ts`, `packages/game-server/src/infrastructure/api/app.test.ts`, and `packages/game-server/src/application/services/combat/combat-service-domain.integration.test.ts`.
