Abstract

This report presents a preliminary case study comparing three approaches to implementing features in a deterministic D&D 5e rules engine: (1) a single AI coding agent (GitHub Copilot with Claude Opus 4.6), (2) a multi-agent orchestration system with domain-specialized sub-agents, and (3) an experienced human developer following test-driven development (TDD) practices. Two implementation tasks of escalating complexity were evaluated: a moderate-complexity cover mechanics feature and a high-complexity class feature set (Barbarian Phase 8.1 — six sub-features spanning domain rules, application services, and infrastructure adapters).

Key Findings: Both AI approaches reduce wall-clock development time by approximately 12–32× relative to the human baseline while eliminating compile/run errors entirely. The multi-agent approach completed each task in 1 premium request versus 3–5 for single-agent, and prevented 5 gameplay-breaking defects via pre-implementation SME review.

These findings are preliminary. The study’s small sample size (n=2 tasks, n=1 human developer, n=1 run per condition), ordering bias, and domain specificity preclude generalization. The report is intended for internal evaluation of AI-assisted development workflows.

1. Introduction

1.1 Motivation

Enterprise software teams face a persistent challenge: implementing complex, rule-heavy business logic correctly the first time. Game rules engines — particularly those implementing tabletop RPG systems like Dungeons & Dragons 5th Edition (2024) — represent an extreme case. They require:

  • Deep domain knowledge: Hundreds of interacting rules with subtle edge cases
  • Cross-cutting concerns: A single feature touches domain entities, combat resolution, damage calculation, turn management, action economy, and AI decision-making
  • Deterministic correctness: Rules must be implemented exactly as specified; “close enough” creates compounding errors

1.2 System Under Study

The target system is a TypeScript/Node.js game server implementing D&D 5e 2024 rules as a deterministic engine. Key architectural characteristics:

  • Domain-Driven Design: Pure rules logic in domain/, orchestration in application/, adapters in infrastructure/
  • Dual-path patterns: Multiple code paths for the same operation (e.g., player-rolled vs. server-auto-rolled initiative)
  • Comprehensive E2E harness: 145+ scenario-driven integration tests with deterministic mock dice

These dual-path patterns are particularly relevant — they create a class of defects where a feature works in one code path but silently fails in another.

1.3 Research Questions

  1. RQ1: How do AI agents compare to a human developer in time-to-completion and defect rate?
  2. RQ2: Does multi-agent orchestration with domain-specialized review reduce defects compared to a single agent?
  3. RQ3: What is the efficiency tradeoff — does multi-agent overhead justify its quality benefits?

4. Discussion

4.1 Why the Human Developer Is Slower

The time differential is not about typing speed. Three structural factors dominate:

Context switching cost: The human must hold the mental model of the entire system while editing one file. AI agents load file contents as needed and read 10+ files in parallel — context switching has near-zero cost.

Compile-run-debug cycle time: 93 compile/run errors represent 93 interruptions to flow state. AI agents generate valid code on the first attempt by having type definitions and patterns in context simultaneously.

Test specification errors: The most consequential finding is 12 bugs caused by incorrect tests. TDD assumes correct tests, but for complex rule systems, the test author faces the same knowledge gaps as the implementer.

4.2 Multi-Agent vs. Single-Agent: When Review Pays Off

Multi-agent caught 5 gameplay-breaking defects that single-agent missed. These fell into two categories:

Dual-path architectural defects (3 of 5): The codebase has multiple code paths for the same operation. Features that modify one path but not the other silently fail. These defects:

  • Pass all tests that exercise only one path
  • Produce correct behavior in common cases
  • Fail silently — no crash, no error, just wrong game mechanics

Edge-case logic defects (2 of 5): Missing castSpell() early return and null guard issues — standard defensive programming concerns.

4.3 The TDD Paradox

The human developer’s TDD practice introduced more bugs, not fewer, in the high-complexity task. The mechanism: tests encoded wrong assumptions about D&D 5e rules and codebase interaction patterns. Code that passed these flawed tests exhibited incorrect behavior in practice.

This is not an argument against TDD. It is an observation that TDD’s effectiveness is bounded by the test author’s domain knowledge.

5. Limitations

This study has significant methodological limitations that preclude generalization.

LimitationImpact
Sample size (n=2 tasks, n=1 run)No statistical analysis possible; results could be driven by random variance
Single human developerPrimary codebase author = best-case human performance; self-reported metrics subject to recall bias
Ordering biasSingle-agent always ran first; task specs may have been inadvertently improved
AI wall-clock estimatesNot directly measured; order-of-magnitude only
Domain specificityDeterministic rules engine with documented rules — may disproportionately favor AI agents
Cost model simplificationIgnores token consumption, rate limiting, orchestrator setup cost, task spec writing time
No downstream validation“Zero bugs” means zero detectable by current tests — not zero in absolute terms
Hawthorne effectHuman developer knew their work was being compared

Suggested Follow-Up Studies

  1. Increase sample size (5+ tasks × 3+ runs per condition)
  2. Randomize ordering across conditions
  3. Include 3–5 developers of varying experience levels
  4. Instrument AI sessions with precise wall-clock timestamps
  5. Track defect rates over weeks/months of downstream use
  6. Replicate on non-game-rules codebases (financial, medical)
  7. Full cost-benefit analysis including setup and spec-writing time
